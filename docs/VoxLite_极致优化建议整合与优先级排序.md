# VoxLite 极致优化建议整合与优先级排序

## 1. 文档目的

本文基于 `.temp/极致优化建议.md` 中实际合并的 3 份建议，结合当前仓库实现、VoxLite 的产品边界，以及 2026 年 Apple 本地模型能力现状，输出一份**去重、筛选、排序后的可执行优化清单**。

本文目标不是罗列“所有可能的高级玩法”，而是给当前项目提供一份**更贴近现状、可分阶段落地、优先改善主链路体验**的方案列表。

---

## 2. 当前项目判断基线

结合现有代码与文档，当前项目已有以下事实基础：

- 已使用 `SpeechAnalyzer` 端侧转写链路，并在 `Sources/VoxLiteCore/SpeechTranscriber.swift` 中实际处理 `AssetInventory` 模型安装。
- 主链路已经稳定收敛为 `录音 → 转写 → 清洗 → 注入`，实现集中在 `Sources/VoxLiteCore/VoicePipeline.swift`。
- 已支持本地 / 远端 STT 与 LLM 切换，不是“纯本地固定实现”，但产品主张仍然是**端侧优先、轻量、低打扰**。
- 清洗层当前已支持技能模板 + Foundation Model / 远端 LLM，但 fallback 规则仍较轻，场景识别也仍以 `bundleId` 粗分类为主。
- 项目已经有性能目标与质量门禁：P50 < 1s、P95 < 1.8s、空闲内存 < 150MB、空闲 CPU < 3%。

这意味着：

1. 当前最值钱的优化，不是继续扩功能面，而是**缩短冷启动、降低主链路波动、提升清洗质量和注入稳定性**。
2. Apple 本地模型更适合承担**轻量到中等规模、低延迟、隐私优先的处理任务**，不适合在当前阶段强行扩成“大而全的端侧智能平台”。
3. 一切建议都应优先服务于当前定位：**极简原生 macOS 语音输入工具**，而不是平台化语音工作台。

---

## 3. 去重与筛选原则

本次整合采用以下原则：

- **保留高相关项**：直接改善录音、转写、清洗、注入、权限引导、模型资源管理、感知延迟的建议。
- **合并相似项**：例如“预热 SpeechAnalyzer”“prepareToAnalyze 预加载”“焦点变化提前唤醒”统一并入“模型预热与冷启动治理”。
- **降权未来项**：例如完整 InputMethodKit 输入法化、Top-3 候选系统、持续多轮智能交互、复杂产品化工作台能力。
- **剔除低相关项**：例如 LoRA 适配器、本地 RAG 学习用户文件、通用 Freemium 商业设计、speaker diarization、多人会议能力。

---

## 4. 排序方法

排序依据为三项综合判断：

- **实施难度**：越容易落地越靠前。
- **性能收益**：是否明显改善冷启动、P50/P95、CPU/内存、主链路波动。
- **体验收益**：是否明显改善“按住说、松开写”的可感知速度、稳定性和结果可用率。

排序时额外考虑两条约束：

1. 如果某项是其他优化的前置能力，则优先级会上移。
2. 如果某项虽然想象空间大，但明显偏离当前产品边界，则优先级下调。

---

## 5. 排序后的建议清单

### P0：优先立即推进

#### 1. 模型预热与冷启动治理

- **建议内容**：在当前主链路之外补一层可控预热机制，围绕 `SpeechAnalyzer` / `SpeechTranscriber` 的首次就绪时间做治理，而不是等用户按下热键才进入完整初始化。
- **合并来源**：
  - `prepareToAnalyze` 提前预热
  - Focus Change 时预加载模型
  - 降低首字延迟
- **为什么保留**：这是 3 份建议里重复度最高、且和现有项目最贴合的一类建议。VoxLite 的核心竞争力之一就是“松手就写”，冷启动抖动会直接伤害体验。
- **结合当前项目的落点**：
  - `SpeechTranscriber.swift` 已经使用 `SpeechAnalyzer`，适合在现有实现上追加预热与缓存策略。
  - 可以先做轻量预热，不必立即引入复杂模型管理子系统。
- **实施难度**：中
- **性能收益**：高
- **体验收益**：高

#### 2. Asset/模型就绪状态显式化

- **建议内容**：把“模型未下载 / 模型安装中 / 模型未就绪 / 可用”从隐式失败改成显式状态，并在首次使用或模型缺失时给出清晰引导。
- **合并来源**：
  - `AssetInventory` 动态下载与安装逻辑
  - 模型不可用时的清晰引导
  - Apple 本地模型 readiness 管理
- **为什么保留**：当前项目已经在代码层处理模型安装，但用户侧的“为什么现在不能转写/为什么需要等待”仍然容易变成黑盒体验。
- **结合当前项目的落点**：
  - `AppViewModel` 已有 `speechStatus` 和 `foundationModelStatus`，适合继续扩成更明确的资源状态说明。
  - 与现有 onboarding / 失败回退体系天然兼容。
- **实施难度**：低
- **性能收益**：中
- **体验收益**：高

#### 3. 把性能优化从总时延推进到“阶段级优化”

- **建议内容**：继续沿用现有 `VoicePipeline` 埋点，但把优化行动聚焦到分阶段：录音停止、转写、清洗、注入分别观察和压缩，而不是只盯总 P50/P95。
- **合并来源**：
  - 低延迟优化
  - 运行时 Profiling
  - 用户感知速度优先于单纯模型能力
- **为什么保留**：当前代码已经有阶段埋点与 percentile 能力，这是最容易马上形成正反馈的一项工程优化。
- **结合当前项目的落点**：
  - `VoicePipeline.swift` 已暴露阶段 percentile 方法。
  - `VoxLiteSelfCheck` 已经验证各阶段延迟门槛，适合直接升级为更面向优化决策的指标展示。
- **实施难度**：低
- **性能收益**：高
- **体验收益**：中高

#### 4. 清洗层的“轻量语义重构”增强

- **建议内容**：把当前清洗从“模板生成 + 规则兜底”提升到更稳定的口语清洗，包括断句、去口癖、轻量语义补全、标点与大小写规范化，但仍然保持可预测、低延迟。
- **合并来源**：
  - AFM 延迟精修
  - 口语到结构化意图文本
  - 去口癖 / 断句 / 语义补全
- **为什么保留**：当前 `RuleBasedTextCleaner` 的 fallback 很轻，这正是项目最明显的质量短板之一。
- **结合当前项目的落点**：
  - 不建议一开始追求“大模型智能重写”，而应优先提升当前 fallback 的可用度。
  - 先增强 `communication / development / writing` 三路由下的规则与 prompt 质量，比扩新功能更划算。
- **实施难度**：中
- **性能收益**：中
- **体验收益**：高

---

### P1：高价值的第二阶段

#### 5. 模型保留与释放策略分层

- **建议内容**：根据设备能力与当前工作状态，设计分层的 model retention 策略，在更强的 Apple Silicon 设备上尽量保活，在资源紧张或非理想机型上及时释放。
- **合并来源**：
  - `ModelRetention` 分级策略
  - Intel 与 Apple Silicon 区别对待
  - 功耗与内存精细化管理
- **为什么保留**：项目 README 与 PRD 明确要求同时兼顾 Apple Silicon 与 Intel；这是现实约束，不是锦上添花。
- **结合当前项目的落点**：
  - 适合建立在“预热与就绪状态显式化”之后推进。
  - 重点不是做复杂控制台，而是围绕冷启动、驻留内存和空闲 CPU 做设备分层策略。
- **实施难度**：中高
- **性能收益**：高
- **体验收益**：中高

#### 6. 场景感知从 `bundleId` 粗分类升级为“上下文增强清洗”

- **建议内容**：在不引入复杂 RAG 的前提下，补充更丰富的上下文信息进入清洗阶段，例如输入目标类型、历史风格、当前模式、局部词汇偏置。
- **合并来源**：
  - `AnalysisContext` 注入
  - 动态 prompt / grammar 切换
  - 热词注入 / 术语识别增强
- **为什么保留**：这是对当前 `ContextResolver.swift` 最直接、最现实的增强方向。
- **结合当前项目的落点**：
  - 当前 `ContextInfo` 结构已经存在，可以逐步扩字段。
  - 重点优先放在开发场景术语、常见命名和沟通/写作风格差异，而不是做大而全的个性化系统。
- **实施难度**：中
- **性能收益**：中
- **体验收益**：高

#### 7. 注入能力从单实现提升为“策略层”

- **建议内容**：保留当前剪贴板替身为默认路径，但为更多注入策略预留协议和配置层，例如 direct typing 或按应用的兼容性策略。
- **合并来源**：
  - 系统级文本插入优化
  - 输出控制层
  - 多应用兼容性注入策略
- **为什么保留**：当前 `ClipboardTextInjector` 很适合 MVP，但这类工具的长期稳定性瓶颈往往最终出在“不同 App 的注入兼容性”。
- **结合当前项目的落点**：
  - 当前结构已基于协议抽象，适合逐步演进。
  - 建议先做策略抽象，不急着一次性扩很多方式。
- **实施难度**：中
- **性能收益**：低到中
- **体验收益**：高

#### 8. 更明确的处理中反馈，而不是更重的 UI

- **建议内容**：优化 Processing 阶段反馈，让用户知道系统是在“转写”“清洗”“等待注入”还是“已回退剪贴板”，但避免引入过重的 overlay 或复杂交互。
- **合并来源**：
  - 骨架屏 / 流光边框
  - 实时反馈降低等待焦虑
  - 用户感知速度优化
- **为什么保留**：对这类短链路产品来说，感知速度和真实速度同样重要。
- **结合当前项目的落点**：
  - 当前 `AppViewModel` 已有 `stateText`、`speechStatus`、`foundationModelStatus`、`resourceHint`，已经具备继续优化的基础。
  - 不建议引入复杂视觉系统，保持轻量更符合项目定位。
- **实施难度**：低到中
- **性能收益**：低
- **体验收益**：中高

---

### P2：有价值，但应延后或小范围试点

#### 9. 开发场景专用增强（命名、提交语气、代码指令）

- **建议内容**：围绕开发场景继续强化现有能力，例如 commit message、注释、命名风格转换等，而不是直接做 AST 级代码生成。
- **合并来源**：
  - code-aware 输入
  - Smart Paste
  - 开发者场景专项优化
- **为什么保留**：项目 PRD 已经明确把开发场景列为核心场景之一，且当前默认技能里也已有 prompt 导向能力。
- **为什么不排更前**：如果现在直接冲向“说一句话生成代码结构”，实现复杂度和误判风险都会明显增大。
- **实施难度**：中
- **性能收益**：低
- **体验收益**：中高

#### 10. 个性化词典 / 高频短语缓存

- **建议内容**：做轻量本地用户词典和高频短语缓存，用于专有名词、常见短语、个体表达习惯修正。
- **合并来源**：
  - 个性化语言模型
  - 本地 user dictionary
  - 高频短语缓存
- **为什么保留**：这类能力符合端侧优势，且比本地 RAG 安全、轻量、可控。
- **为什么不排更前**：当前项目最先要解决的是系统性延迟与基础清洗质量，不是个性化壁垒。
- **实施难度**：中
- **性能收益**：低到中
- **体验收益**：中

#### 11. 更细粒度的 VAD / 端点检测治理

- **建议内容**：优化断句、录音结束判断与超短录音处理逻辑，降低“松手太快”“停顿误判”“空录”带来的体验损失。
- **合并来源**：
  - VAD 接管
  - SpeechDetector / 端点检测
  - 自动启停建议
- **为什么保留**：这类建议对感知延迟和误触容忍度有帮助。
- **为什么不排更前**：当前产品是“按住说、松开写”，交互上已经天然缩小了自动端点检测的重要性。
- **实施难度**：中高
- **性能收益**：中
- **体验收益**：中

---

### P3：当前阶段不建议优先投入

#### 12. 完整 InputMethodKit / 系统输入法化改造

- **保留为远期方向，不纳入近期优先级。**
- **原因**：当前产品明确是 Menu Bar App 形态，现有主链路已经可工作。完整输入法化会显著增加系统复杂度、权限与维护成本，不适合作为当前阶段的优化主轴。

#### 13. Top-3 候选、多结果选择器、复杂确认流

- **保留为中后期交互探索，不纳入当前高优先级。**
- **原因**：这会显著拉长交互链路，和 VoxLite 当前“低认知、低打扰”的原则并不完全一致。

#### 14. 本地 RAG 学习用户文件风格

- **当前不建议推进。**
- **原因**：虽然“端侧个性化”听起来很强，但实现复杂、隐私解释成本高、收益不稳定，也明显超出当前仓库能力边界。

#### 15. LoRA、复杂本地模型适配器、超大端侧模型路线

- **当前剔除。**
- **原因**：Apple 本地模型在当前阶段更适合轻量、低延迟、工具型能力，不适合把项目重心放在复杂模型训练/适配体系上。

#### 16. Freemium、多人说话标注、会议级能力、跨场景平台化

- **当前剔除。**
- **原因**：这些建议更偏产品扩张，不属于当前“极致优化主链路”的范畴。

---

## 6. 去重后的最终结论

如果只保留**当前阶段最值得做的 5 项**，建议顺序如下：

1. **模型预热与冷启动治理**
2. **Asset/模型就绪状态显式化**
3. **阶段级性能优化与指标驱动治理**
4. **清洗层轻量语义重构增强**
5. **场景感知与上下文增强清洗**

这 5 项同时满足以下条件：

- 直接服务现有 `VoicePipeline`
- 与 Apple 本地模型现实能力匹配
- 对 P50/P95、结果可用率和主观速度都有明确帮助
- 不会把项目从“轻量语音输入工具”拉偏到“平台化语音工作台”

---

## 7. 建议的实施节奏

### 第一阶段：先把主链路做稳

- 模型预热与冷启动治理
- 模型就绪状态显式化
- 阶段级性能剖析与看板化

### 第二阶段：再把结果质量做强

- 清洗层轻量语义重构增强
- 场景上下文增强
- 开发场景专项增强

### 第三阶段：最后做兼容性与个性化

- 注入策略层
- 轻量用户词典 / 高频短语缓存
- 更细粒度端点检测治理

## 8. 本次被合并或降权的代表项

| 原始方向 | 处理结果 | 原因 |
|---|---|---|
| `prepareToAnalyze`、模型预唤醒、焦点变化预加载 | 合并 | 本质都是冷启动治理 |
| `AnalysisContext`、热词注入、动态 prompt | 合并 | 本质都是上下文增强清洗 |
| AFM 延迟精修、语义重构、断句去口癖 | 合并 | 本质都是清洗层质量增强 |
| Smart Paste、输出控制层、系统级插入优化 | 合并 | 本质都是注入策略与输出行为治理 |
| Top-3 候选、多路径结果 | 降权 | 会拉长交互链路，不符合当前 MVP 心智 |
| InputMethodKit 改造 | 降权 | 工程面过大，偏离当前阶段 |
| 本地 RAG / LoRA / 超大模型能力 | 剔除 | 超出当前项目边界，也不符合 Apple 本地模型现实投入产出比 |

## 9. 最终判断

这三份“极致优化建议”里，真正对 VoxLite 当前阶段最有价值的，不是那些听起来最炫的方向，而是几类更朴素但更关键的系统工程能力：

- **更快进入可识别状态**
- **更稳定地让用户知道系统正在做什么**
- **更可靠地把口语转成可直接使用的文本**
- **更少波动地完成注入与回退**

对于当前项目来说，真正的“极致优化”不是继续扩功能面，而是把这条已经成型的主链路继续打磨到更快、更稳、更可预期。
